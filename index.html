01

Structured

A disciplined build process from idea to production.

01 · Structured

Think before you type

  • Start with ideation. Define the problem, the user, and the core value before a single line of code.
  • Break the product into shippable features. Each one is a self-contained unit with a clear definition of done.
  • Write the spec before the code. What does it do? What are the edge cases? What does success look like?
  • Review the spec first. Catch ambiguity and wrong assumptions at the cheapest possible moment.

01 · Structured

Ship like you mean it

  • Write the code, then review it. Check for correctness, security, and simplicity — in that order.
  • Write and run tests before merging. Tests are how you sleep at night when you ship on Fridays.
  • Main is only reached through CI/CD. Automated checks gate every deployment — no exceptions, including for you.

02

Secure

Security as a foundation, not an afterthought.

02 · Secure

Lock down the basics

  • Store secrets in .env + env vars. Never hardcode API keys, database URLs, or credentials in source code.
  • Add .env to .gitignore before the first commit. Treat any exposed secret as compromised — rotate immediately.
  • Never store plaintext passwords. Use bcrypt or argon2, or delegate auth to Clerk, Supabase Auth, or Auth0.
  • Sanitize user input before it touches a prompt. Prompt injection is the new XSS — treat untrusted text like untrusted text.

02 · Secure

Defense in depth

  • Return only what the frontend needs. Never expose hashed passwords, internal IDs, or sensitive fields in API responses.
  • Validate inputs on the server. Client-side validation is UX — it is not security.
  • Every route checks authn and authz. "Are you logged in?" and "Are you allowed to do this?" are two different questions.
  • Implement row-level security. Users can only ever see and mutate their own data — enforced at the database.

03

Scalable

Performance and growth without overengineering.

03 · Scalable

Build for growth, not for ghosts

  • Design APIs to be extensible. Clean versioning, consistent response shapes, clear separation of concerns.
  • Multi-tenant from day one. The same infrastructure should serve many customers without duplication.
  • Optimize where it matters. Cache repeated LLM calls, queue slow work, paginate large lists.
  • Don't write code you don't need. Every line is a liability. Solve the problem in front of you — not hypothetical futures.
  • Prefer managed infrastructure. Databases, queues, hosting — pay someone else to be on call early on.
