01

Structured

A disciplined build process from idea to production.

01 · Structured · ground floor

GitHub in 60 seconds

Everyone is an engineer now.

  • Repository. A folder for your project that remembers every change you make.
  • Branch & main. A branch is a working copy where you make changes in isolation. main is the source of truth — the version users actually see.
  • Commit. A recorded change paired with a short note. Builds an audit trail you can review or roll back to.
  • Pull Request (PR). A formal proposal to merge changes into main. Reviewed and approved by the team before anything goes in.
  • CI/CD. Continuous Integration / Continuous Delivery — the automated pipeline that tests every change and deploys approved code to users.

01 · Structured

Think before you type

  • Start with ideation. Define the problem, the user, and the core value before a single line of code.
  • Break the product into small features you can release one at a time. Each one has a clear goal and a clear finish line.
  • Write the spec before the code. What does it do? What are the edge cases? What does success look like?
  • Review the spec first. Catch ambiguity and wrong assumptions at the cheapest possible moment.

01 · Structured

Ship like you mean it

  • Write the code, then review it. Check for correctness, security, and simplicity — in that order.
  • Write and run tests before merging. Tests are how you sleep at night when you ship on Fridays.
  • Code only reaches main through CI/CD. Every change passes automated checks before it ships — no exceptions, including for you.

02

Secure

Security as a foundation, not an afterthought.

02 · Secure

Lock down the basics

  • Keep secrets out of your code. API keys, database passwords, anything sensitive — store them in a separate config file the app loads at runtime.
  • Tell Git to ignore the secrets file before your first commit. If a key ever leaks into history, treat it as compromised and rotate it the same day.
  • Never save passwords as plain text. Hash them with a one-way function (bcrypt or argon2), or delegate login entirely to a provider like Clerk or Auth0.
  • Don't paste user input straight into a prompt. Attackers can hide instructions inside it — called "prompt injection." Validate and clean it first.

02 · Secure

Defense in depth

  • Only send the data your app actually needs. Hide passwords, internal IDs, and sensitive fields from every API response.
  • Always re-check user input on the server. The form check in the browser is a nicety, not a defense — anyone can bypass it.
  • Every route checks authn and authz. "Are you logged in?" and "Are you allowed to do this?" are two different questions.
  • Implement row-level security. Users can only ever see and mutate their own data — enforced at the database.

02 · Secure · a word of caution

Auto-approve isn't a free pass

These flags hand the keys to the model. Use them in a room you can afford to blow up.

  • Run it in a throwaway environment. A disposable container, scratch VM, or isolated folder — somewhere you can blow up without losing real work.
  • Keep production credentials away from it. No live database, no real API keys, no long-lived access tokens in the environment.
  • Don't let it install random code. Lock down the list of dependencies and review anything new before it runs.
  • Pin dependencies to specific versions. "Latest" is how a compromised package update becomes your security incident overnight.
  • Don't let it delete things you can't recover. Block destructive commands (wiping tables, wiping files) behind a human check.

03

Scalable

Performance and growth without overengineering.

03 · Scalable

Build for growth, not for ghosts

  • Design APIs to be extensible. Clean versioning, consistent response shapes, clear separation of concerns.
  • Multi-tenant from day one. The same infrastructure should serve many customers without duplication.
  • Optimize where it matters. Cache repeated LLM calls, queue slow work, paginate large lists.
  • Don't write code you don't need. Every line is a liability. Solve the problem in front of you — not hypothetical futures.
  • Prefer managed infrastructure. Databases, queues, hosting — pay someone else to be on call early on.
